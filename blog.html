Flexin On 'Em With The Flexbox

Today was definately my favorite day at The Iron Yard. We really got to utiilize flexbox and dive into positions (i.e. relative and absolute). Let me say, after dealing with Bootstrap for the better part of 4 months now, flexbox is the effing shit! It makes styling a site for responsive behavior a breeze. We built the front of end two websites using flexbox. Hell, I even had a flexbox, in a flexbox, in a flexbox....That's meta bro.

Flexbox works by assigning a container element with the display value of 'flex'. This allows for all child elements of said container element to become flex-items. These flex items can then easily be placed in a row or column, in reverse order, and they can even be given spacing around them to make the child elements apspear clean and professional. This is honestly just scratching the surface of what CSS's native grid system, Flexbox, is capable of.

I can say that I will ikely not be utilizing bootstrap any time soon, as I think I found my new grid system GF. Let's hope we stay together for a while.

Today, we went over routes vs. paths, rendering vs. redirecting, cookies, and sessions. Routes are simply the route a devloper wants to send or get data through, while the path contins a specific destination. Stackoverflow defines rendering and redirect like the following: "res.render() will render the view with data passed to it, res.redirect() will redirect a user to another page (at which point the request starts over)". Where 'res' would be the response from the server. Cookies save the user's session data, are statefull (have data), and reside on the client (a user's computers'). A session is the period of time when the user is on an application and reside on the server. Devlopers will typically set a maximum length of time a session can be live (i.e. 15 minutes).

Our project for the day was to create an express server application that would allow a user to signup, login, and then view their custom login page based on server-side authorization. Corresponding mustache files were made for each. At a high level, the public folder containing all the mustache and css files is set statically, a user then goes to the home page, "/", clicks either signup or login, is then routed to eaither, "/signup" or "/login", fills out a form, the form sends data back to the server through either the "/users" or "/login" route, and then are taken to their personal profile page "/profile", where they prompted with a custom greeting message.

We first dove into SQL and learned the basic SQL commands, SELECT, INSERT, DELETE, FROM, WHERE, and JOIN. SELECT queries are used when retrieving data from a database. INSERT statements are used for inserting data into a database. DELETE statements are used to delete data from a database. The FROM paramater specifies which table you are adding to or manipulating data from. WHERE acts as a filter for your query. JOIN allows developers to access tables from other tables by using primary keys and foreign keys. After getting the basics of SQL down, we went into learning an ORM. ORM stands for Object-Relational Mapping. At a high level, Sequelizer makes sense, but getting into the syntax of it was rather confusing to me. It's pretty damn cool that I am able to manipulate a database with Javascript. If my server is running on Node.js, and I am using sequelizer, then my server configuration and database queries are both using Javascipt. This makes my backend code both efficient and fluid. Setting up Sequlizer, like most ORM's, is a bit of a pain, but once it's initialized and configured, data manipulation isn't too difficult. Although, the documentation is a MUST for any ORM. MongoDB, Primary Keys, & Foreign Keys

This week we have been going over different types of databases other than relational databases or SQL databases. Relational databases work off of a schema and have tables. They are structured and as such work with a structured query language like SQL. Databases that are not structured are referred to as no-sql databases. We have been learning MongoDB this week which is an example of a no-sql database. Instead of tables, the data is stored in what is like one massive document. Data does not have to have specific columns. The data going into a no-sql database can be unique to itself.

Today we refactored a project that was simply pulling in data from JSON file to be pulling data in from MongoDB. I imported the JSON file to database that was created via Mongo, and used vanilla mongo syntax to pull data from the DB into a rendered page of my choosing. I was doing server-side rendering using mustache and the data from the mongodb.

So far, I seem to be drawn to Mongo more than Postrgres. I like that the data in Mongo does not have to be structurerd. If you want to retrieve any data from a no-sql database, you can query it, just like you can a relational database. To me, no-sql databases seem to make the most sense unless you are woeking with an enterprise who has a great deal of red tape and a strong technological architecture.

So That's React

I have been curious about using for the better part of 7 months now. I was going to introduce myself to React sooner, but I really wanted to have a solid foundation of Javascript before getting into an exensive Javascript library. It was important to me to have a true understanding of how Javascript works and understand all the quirks that belong to Javascript. Let me say, I definately glad I did.

So react uses components to structure applications. Every part of the webpage is a component. The header, the navbar, the sidebar, the footer, everything is a component. This is the best part about react. React uses a thing called a 'key' to keep track of every component and every child in a component. React only renders the part of the site that has changed. This makes react effective, compartmentalized, and fast. React is ideal for websites that are passing alot of data through it. Since react is only updating the components that change, the nodes that are getting new data are the only thing being repainted on the page. React does this by using what is called the 'virtual DOM'. React is constantly comparing the virtual DOM to the actual DOM and if react detects a difference between the two, the repaints the node that changed.

The weird par about react is that it uses JSX. JSX is basically Javascript + XML/HTML. In react, you write html in javascript files. Since you are in a javascript file, you can write Javascript. This makes for a very interesting syntax. React uses state and props to pass data around. State is data that belongs to a particular component, a smart component. Props is data that is passed to a 'dumb component' from it's parent component. It is ideal to have many dumb components and few smart components.

I have built a couple of applications using react, and can say I really do like it. Working with react is like working with custom legos. You build lego pieces (components) that you then put together to make a giant lego structure. I am still new to react, but I am really trying to dive into it and understand as much as possible.

React & Redux

This past week, I have spent working with and learning Redux. Redux is a state management module that allows developers to keep track of state in one place instead of having state spread across different components. Redux does this through a 'store'. A store in redux is simply a place where state lives. If you want data from the store, you dispatch it to the component you want to use it in. Actually, you technically 'map' state and 'dispatch' actions.

Actions in redux are like messages. These actions then get sent to what is called a reducer. The reducer is essentially a function that manipulates data sent over through an action. An action has a payload that is sent to the reducer along with the action itself. A payload is data that is sent through an action to the reducer. Actions are typically used via clicks on buttons. The payload is typically data from an input or form. There are also containers, which hold the components. As mentioned earlier, state is mapped to compnents from the store and actions are dispatched to components. Both of which are done through functions at the bottom of the container or the component itself.

I actually enjoy working with redux and how easy it makes passing state from component to component. The problem with redux is getting the architecture right and remembering to always to clone state through each of the cases in the reducers. Bottom-line, Redux rules, but it truly only makes sense when you are working on a large application with mass amounts of data. I keep hearing about Mobx, which is a simpler approach to state management and WAYYY less cumbersome to set up. I'm stoked to have a use case for Mobx in the near future.
